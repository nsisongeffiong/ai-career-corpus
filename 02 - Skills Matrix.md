# Skills Matrix

Maps every corpus-evidenced skill to years, recency, proficiency, and resume-safe phrasing. Build it from the Career Spine, Wins Bank, Project Portfolio, and any archived resumes — then **validate it yourself**; mark remaining estimates with `~`.

> Proficiency scale: 1 = familiar, 2 = assisted use, 3 = independent, 4 = strong/go-to person, 5 = expert/could teach it.
>
> **Usage rule for generation sessions:** only include a skill on a resume if (a) it appears in this matrix, AND (b) the JD asks for it or an adjacent term. Respect the "Do not overclaim" column — it is binding.

## Why the "Do not overclaim" column exists

Old JD-tailored resumes inflate. A skill you listed once for ATS reasons is not a skill you have. This column records the honest ceiling per skill so no generation session ever has to guess — and the **Known gaps** section at the bottom records what recent JDs asked for that you simply don't have. An honest gap flagged in a draft beats a fabricated claim that dies in the interview.

Two patterns worth stealing:

- **The honest bridge.** When a JD demands a tool you lack but you have deep experience in the *same problem space* with a different tool, record the bridge explicitly (e.g. "JDs asking for policy-tool X: claim only our declarative-governance work with tool Y — same problem space, different tool — and never claim X itself"). The bridge phrasing is pre-authorized; the fabrication is pre-banned.
- **Attribution locks.** When a metric or tool belongs to one era/employer and could plausibly be misfiled under another, pin it: "attribute to [Role A], never [Role B]." Misattributed wins fail reference checks.
